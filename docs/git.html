39.1 ¿Por qué usar Git y GitHub?

Hay tres razones principales para usar Git y GitHub.

  1. Compartir: incluso si no aprovechamos la funcionalidad avanzada y poderosa de control de versiones, aún podemos usar Git y GitHub para compartir nuestro código. Ya hemos mostrado cómo podemos hacer esto con RStudio.

  2. Colaboración: una vez que configura un repositorio central, puede hacer que varias personas realicen cambios en el código y mantengan las versiones sincronizadas. GitHub ofrece un servicio gratuito para repositorios centralizados. GitHub también tiene una utilidad especial, llamada pull request, que cualquiera puede usar para sugerir cambios en su código. Puede aceptar o rechazar fácilmente la solicitud.

  3. Control de versiones: las capacidades de control de versiones de Git nos permiten realizar un seguimiento de los cambios que realizamos en nuestro código. También podemos volver a las versiones anteriores de los archivos. Git también nos permite crear branches en los que podemos probar ideas, luego decidir si fusionamos la nueva rama con la original.

Aquí nos centramos en los aspectos de uso compartido de Git y GitHub y remitimos al lector a los enlaces anteriores para obtener más información sobre esta poderosa herramienta.

39.2 Cuentas GitHub

Después de instalar git127, el primer paso es obtener una cuenta de GitHub. Las cuentas básicas de GitHub son gratuitas. Para hacer esto, vaya a [GitHub] (github.com) donde verá un cuadro en el que puede registrarse.

Desea elegir un nombre con cuidado. Debe ser breve, fácil de recordar y deletrear, relacionado de alguna manera con su nombre y profesional. Este último es importante ya que podría estar enviando a posibles empleadores un enlace a su cuenta de GitHub. En el siguiente ejemplo, estoy sacrificando la facilidad de deletrear para incorporar mi nombre. Sus iniciales y apellidos suelen ser una buena opción. Si tiene un nombre muy común, es posible que tenga que tenerlo en cuenta. Una solución simple sería agregar números o deletrear parte de su nombre.

La cuenta que uso para mi investigación, rafalab, es la misma que uso para mi página web128 y Twitter129, lo que hace que sea fácil de recordar para los que siguen mi trabajo

Una vez que tenga una cuenta de GitHub, estará listo para conectar Git y RStudio a esta cuenta.

Un primer paso es dejar que Git sepa quiénes somos. Esto facilitará la conexión con GitHub. Comenzamos abriendo una ventana de terminal en RStudio (recuerde que puede obtener una a través de Herramientas en la barra de menú). Ahora usamos el git config comando para decirle a Git quiénes somos. Escribiremos los siguientes dos comandos en nuestra ventana de terminal:

git config --global user.name "Your Name"
git config --global user.mail "your@email.com"

Debe usar la cuenta de correo electrónico que utilizó para abrir su cuenta de GitHub. La sesión de RStudio debería verse así:

Empieza yendo a las Opciones globales, seleccionando Git/ SVN, y luego ingresas una ruta para el ejecutable de Git que acabamos de instalar.

En la instalación predeterminada de Windows, esto será C:/ Program File/ Git/ bin/ git.exe, pero debe encontrarlo explorando su sistema, ya que esto puede cambiar de un sistema a otro. Ahora, para evitar ingresar nuestra contraseña de GitHub cada vez que intentemos acceder a nuestro repositorio, crearemos lo que se llama SSH RSA Key. 39.3 Repositorios de GitHub

Ahora está listo para crear un repositorio de GitHub (repositorio). La idea general es que tendrá al menos dos copias de su código: una en su computadora y otra en GitHub. Si agrega colaboradores a este proyecto, cada uno tendrá una copia en su computadora. La copia de GitHub generalmente se considera la copia master con la que se sincroniza cada colaborador. Git te ayudará a mantener sincronizadas todas las copias diferentes.

Como se mencionó, una de las ventajas de mantener el código en un repositorio de GitHub es que puede compartirlo fácilmente con posibles empleadores interesados en ver ejemplos de su trabajo. Debido a que muchas compañías de ciencia de datos usan sistemas de control de versiones, como Git, para colaborar en proyectos, también pueden estar impresionadas de que ya conozca al menos los conceptos básicos.

El primer paso para crear un repositorio para su código es inicializar en GitHub. Debido a que ya creó una cuenta, tendrá una página en GitHub con la URL http://github.com/username.

Para crear un repositorio, primero inicie sesión en su cuenta haciendo clic en el botón Iniciar sesión en [https://github.com] (http://github.com). Es posible que ya haya iniciado sesión, en cuyo caso el botón Iniciar sesión no aparecerá. Si inicia sesión, deberá ingresar su nombre de usuario y contraseña. Le recomendamos que configure su navegador para recordar esto para evitar escribirlo cada vez.

Una vez en su cuenta, puede hacer clic en Repositorios_y luego hacer clic en_Nuevo para crear un nuevo repositorio:

Entonces querrá elegir un buen nombre descriptivo para el proyecto. En el futuro, es posible que tenga docenas de repositorios, así que tenga esto en cuenta al elegir un nombre. Aquí usaremos homework-0. Le recomendamos que haga público el repositorio. Si desea mantenerlo en privado, deberá pagar un cargo mensual.

Ahora tienes tu primer repositorio en GitHub. El siguiente paso será clonarlo en su computadora y comenzar a editar y sincronizar usando Git.

Para hacer esto, es conveniente copiar el enlace proporcionado por GitHub específicamente para conectarse a este repositorio, usando Git como se muestra a continuación. Más tarde necesitaremos copiar y pegar esto, así que asegúrese de recordar este paso.

39.4 Descripción general de Git

Las principales acciones en Git son:

  1. pull cambios desde el repositorio remoto, en este caso el repositorio de GitHub
  2. agregar archivos, o como decimos en los archivos stage de la jerga de Git
  3. commit cambios al repositorio local
  4. push cambios al repositorio remote, en nuestro caso el repositorio GitHub

Ahora vamos a hacer cambios en estos archivos. Eventualmente, queremos que estas nuevas versiones de los archivos sean rastreadas y sincronizadas con el repositorio ascendente. Pero no queremos hacer un seguimiento de cada pequeño cambio: no queremos sincronizar hasta que estemos seguros de que estas versiones son lo suficientemente finales como para compartirlas. Por esta razón, el sistema de control de versiones no guarda las ediciones en el área de preparación.

Para demostrarlo, agregamos un archivo al área de preparación con el git add mando. A continuación creamos un archivo usando Unix echo comando solo como ejemplo (en realidad usaría RStudio):

echo "test" >> new-file.txt

También estamos agregando un archivo temporal que no queremos rastrear en absoluto:

echo "temporary" >> tmp.txt

Ahora podemos organizar el archivo que finalmente queremos agregar a nuestro repositorio:

git add new-file.txt

39.5 Inicializando un directorio Git

Ahora aprendamos la segunda forma en que podemos comenzar: inicializando un directorio en nuestra propia computadora en lugar de la clonación.

Supongamos que ya tenemos un directorio local poblado y queremos convertir este directorio en un repositorio colaborativo de GitHub. La forma más eficiente de lograr esto es inicializando el directorio local.

Para demostrar cómo hacer esto, inicializaremos el directorio de asesinatos de armas que creamos en la Sección 38.7. Tenga en cuenta que ya creamos un directorio con varios subdirectorios en nuestra computadora, pero aún no tenemos un repositorio local de Git o un repositorio ascendente de GitHub.

Comenzamos creando un nuevo repositorio en nuestra página de GitHub. Hacemos clic en el botón Nuevo:

Lo llamamos murders aquí para que coincida con el nombre del directorio en nuestro sistema local. Pero si está haciendo esto para otro proyecto, elija un nombre apropiado.

Luego obtenemos una serie de instrucciones sobre cómo comenzar, pero en su lugar podemos usar lo que hemos aprendido. Lo principal que necesitamos de esta página es copiar la URL del repositorio, en este caso: https://github.com/rairizarry/murders.git.

En este momento, podemos comenzar una terminal y cd en nuestro directorio de proyectos locales. En nuestro ejemplo, sería:

cd ~/projects/murders

Luego, inicializamos el directorio. Esto convierte el directorio en un directorio Git y Git comienza a rastrear:

git init

Todos los archivos ahora están solo en nuestro directorio de trabajo; no hay archivos en nuestro repositorio local o en GitHub.

El siguiente paso es conectar el repositorio local con el repositorio de GitHub. En un ejemplo anterior, hicimos que RStudio hiciera esto por nosotros. Ahora tenemos que hacerlo nosotros mismos. Podemos agregar cualquiera de los archivos y confirmarlo:

git add README.txt
git commit -m "First commit. Adding README.txt file just to get started"

Ahora tenemos un archivo en nuestro repositorio local y podemos conectarlo al repositorio ascendente, que tiene url: https://github.com/rairizarry/murders.git.

Para hacer esto, usamos el comando git remote add.

git remote add origin `https://github.com/rairizarry/murders.git`

Ahora podemos usar git push ya que hay una conexión a un repositorio ascendente:

git push

En la sección 40.3 continuamos trabajando con este ejemplo, mientras demostramos cómo podemos usar RStudio para trabajar con Git y mantener un proyecto sincronizado en GitHub.

39.6 Usando Git y GitHub en RStudio

Si bien la línea de comandos Git es una herramienta poderosa y flexible, puede ser algo desalentador cuando estamos comenzando. RStudio proporciona una interfaz gráfica que facilita el uso de Git en el contexto de un proyecto de análisis de datos. Describimos cómo usar esta función RStudio para hacer esto aquí.

Ahora estamos listos para comenzar un proyecto de RStudio que usa control de versiones y almacena el código en un repositorio de GitHub. Para hacer esto, comenzamos un proyecto pero, en lugar de New Directory, seleccionaremos Version Control_y luego seleccionaremos_Git como nuestro sistema de control de versiones:

La URL del repositorio es el enlace que usó para clonar. En la sección 39.3, nosotros usamos https://github.com/username/homework-0.git como ejemplo. En el nombre del directorio del proyecto, debe poner el nombre de la carpeta que se generó, que en nuestro ejemplo será el nombre del repositorio homework-0. Esto creará una carpeta llamada homework-0 en su sistema local. Una vez que haga esto, se crea el proyecto y tiene conocimiento de la conexión a un repositorio de GitHub. Verá en la esquina superior derecha el nombre y el tipo de proyecto, así como una nueva pestaña en el panel superior derecho titulada Git.

Si selecciona esta pestaña, le mostrará los archivos de su proyecto con algunos iconos que le brindan información sobre estos archivos y su relación con el repositorio. En el ejemplo a continuación, ya agregamos un archivo a la carpeta, llamado code.R, que puede ver en el panel de edición.

Ahora debemos prestar atención al panel Git. Es importante saber que sus archivos locales y el repositorio de GitHub no se sincronizarán automáticamente. Como se describe en la Sección 39.4, debes sincronizar usando git push cuando estés listo. Le mostramos que puede hacerlo a través de RStudio en lugar del terminal a continuación.

Antes de comenzar a trabajar en un proyecto colaborativo, generalmente lo primero que hacemos es _pull__ en los cambios desde el repositorio remoto, en nuestro caso el de GitHub. Sin embargo, para el ejemplo que se muestra aquí, dado que estamos comenzando con un repositorio vacío y somos los únicos que hacemos cambios, no necesitamos comenzar tirando.

En RStudio, el estado del archivo en relación con los repositorios remotos y locales se representa en los símbolos de estado con colores. Un cuadrado amarillo significa que Git no sabe nada sobre este archivo. Para sincronizar con el repositorio de GitHub, necesitamos añadir el archivo, luego commitir el cambio a nuestro repositorio de Git local, y luego empujar el cambio al repositorio de GitHub. En este momento, el archivo está en nuestra computadora. Para agregar el archivo usando RStudio, hacemos clic en el cuadro Etapa. Verá que el ícono de estado ahora cambia a una A verde.

Nota: solo estamos agregando el archivo code.R. No necesariamente tenemos que agregar todos los archivos de nuestro repositorio local al repositorio de GitHub, solo los que queremos hacer un seguimiento o los que queremos compartir. Si nuestro trabajo está produciendo archivos de cierto tipo de los que no queremos hacer un seguimiento, podemos agregar el sufijo que define estos archivos al archivo .gitignore. Aquí se incluyen más detalles sobre el uso de .gitignore: [https://git-scm.com/docs/gitignorefont>(https://git-scm.com/docs/gitignore). Estos archivos dejarán de aparecer en su panel RStudio Git. Para el ejemplo que se muestra aquí, solo agregaremos code.R. Pero, en general, para un proyecto RStudio, recomendamos agregar los archivos .gitignore y .Rproj.

Ahora estamos listos para enviar el archivo a nuestro repositorio local. En RStudio, podemos usar el botón Commitir. Esto abrirá una nueva ventana de diálogo. Con Git, cada vez que cometemos un cambio, debemos ingresar un comentario que describa los cambios que se están comprometiendo.

En este caso, simplemente describiremos que estamos agregando un nuevo script. En este cuadro de diálogo, RStudio también le ofrece un resumen de lo que está cambiando al repositorio de GitHub. En este caso, debido a que es un archivo nuevo, todo el archivo se resalta en verde, lo que resalta los cambios.

Una vez que presionamos el botón de confirmación, deberíamos ver un mensaje de Git con un resumen de los cambios que se confirmaron. Ahora estamos listos para empujar estos cambios al repositorio de GitHub. Podemos hacer esto haciendo clic en el botón Push en la esquina superior derecha:

Ahora vemos un mensaje de Git que nos dice que el impulso ha tenido éxito. En la ventana emergente ya no vemos el code.R archivo. Esto se debe a que no se han realizado nuevos cambios desde la última vez que presionamos. Podemos salir de esta ventana emergente ahora y continuar trabajando en nuestro código.

Si ahora visitamos nuestro repositorio en la web, veremos que coincide con nuestra copia local.

¡Felicitaciones, ha compartido código con éxito en un repositorio de GitHub!
